95.I want to fetch data before a compoent is called?
->If we want to fetch data before a component loads, we can use a Resolver in Angular.

A Resolver is a special Angular service that runs before a route is activated. When a user navigates to a route, the resolver is called first, it fetches the required data (like from an API), and only after that, the component is loaded.

This way:
The data is already available when the component loads.
There's no need to show a loading spinner inside the component.
It helps in scenarios where the component contains tables, lists, or important dynamic data.

✅ Summary in Simple Words:
Resolver is used to load data before the component opens. When a route is activated, resolver runs first → fetches data → passes it to the component. This is useful when we want to show things like a table with multiple rows immediately when the component is displayed.

-->An Auth Interceptor is a special service in Angular that automatically adds authentication tokens (like JWT) to outgoing HTTP requests. It helps ensure that all API calls include the necessary authorization credentials without needing to manually add tokens in every request.

->How Does an Auth Interceptor Work?
 ️⃣ Intercepts HTTP requests before they are sent to the server.
 ️⃣ Modifies the request (e.g., adds an Authorization header with a token).
 ️⃣ Forwards the modified request to the next step in the HTTP pipeline.
 ️⃣ Can also handle errors (e.g., redirect to login if the token is invalid).

 ️⃣ Throttling (Simple Example & Explanation)
📌 Definition:
Throttling controls how many times a function runs in a given time. Even if an event happens many times, the function runs at fixed intervals only.

Example 1: Button Click (Without Throttling)
👉 Suppose you have a button, and clicking it many times quickly calls a function every time.

document.getElementById("myButton").addEventListener("click", () => {
  console.log("Button clicked!");
});

❌ If you click 10 times in 1 second, it logs 10 times.


Example 2: Button Click (With Throttling)
👉 Now, let's apply throttling to allow clicking only once every 2 seconds (2000ms).

function throttle(func, delay) {
  let lastTime = 0;
  return function () {
    let now = Date.now();
    if (now - lastTime >= delay) {
      func();
      lastTime = now;
    }
  };
}

const handleClick = throttle(() => {
  console.log("Button clicked!");
}, 2000);

document.getElementById("myButton").addEventListener("click", handleClick);
✅ Now, even if you click 10 times in 1 second, it will only log once every 2 seconds.

Real-Life Use Cases:
✔ Scrolling: Prevents running code too many times when scrolling.
✔ Resizing: Limits function calls when resizing the browser window.
✔ Button Clicks: Stops multiple clicks at the same time.
✔ API Requests: Prevents overloading the server with too many requests.

Summary:
✅ Throttling limits function execution.
✅ Runs at fixed time intervals.
✅ Improves performance and prevents excessive calls.

📌 Event Bubbling (Simple Explanation)
🔹 Event bubbling happens when an event on a child element also triggers the same event on its parent elements.
🔹 Example: If you click a button inside a div, the click event first happens on the button, then moves up to the div.

✅ Example of Event Bubbling:

<div onclick="alert('Div Clicked!')">
  <button onclick="alert('Button Clicked!')">Click Me</button>
</div>

👉 When you click the button:

 ️⃣ "Button Clicked!" appears.
 ️⃣ "Div Clicked!" appears next (because of bubbling).

🛑 How to Stop Bubbling?
👉 If you only want to trigger the button click (not the div), use event.stopPropagation().

<div onclick="alert('Div Clicked!')">
  <button id="myButton">Click Me</button>
</div>

<script>
document.getElementById("myButton").addEventListener("click", function(event) {
  event.stopPropagation(); // Stops bubbling
  alert("Button Clicked!");
});
</script>

✅ Now, only "Button Clicked!" appears when you click the button.

🔹 Key Points:
✔ Event bubbling → Events move up from child to parent.
✔ Stops bubbling → Use event.stopPropagation().
✔ Useful for preventing unwanted event triggers.

9.code coverage
📌 What is Code Coverage?
Code coverage tells you how much of your code is tested by unit tests. If some parts of your code are not tested, they might have bugs.

📌 What is Code Coverage?
👉 Code coverage means checking how much of your code is tested using unit tests.

🔹 How Does It Work?
 ️⃣ When you create a component in Angular, a test file (component.spec.ts) is automatically created.
 ️⃣ You write test cases in the .spec.ts file to check if your code works correctly.
 ️⃣ To check how much of your code is tested, you run:


ng test --code-coverage
 ️⃣ This command runs all test cases and stores the results in the /coverage folder.
 ️⃣ You can open the coverage/index.html file to see which parts of your code are tested and which are missing tests.

🔹 Example
If you have a function in a service:

Local Storage vs. Session Storage (Simple Explanation)
✅ Local Storage

Stores permanent data (until manually cleared).
Data is accessible in all tabs/windows of the same browser.
Common use: Storing login tokens, user preferences, themes, etc.
If you clear local storage, the user gets logged out.
✅ Session Storage

Stores temporary data (deleted when the tab/browser is closed).
Data is only accessible in the current tab.
Common use: Storing temporary session data like form inputs, session tokens, etc.
More secure than local storage because data is cleared automatically when the session ends.
✅ Common Features

Both store up to 5MB of data.

Both are used to store key-value pairs in the browser.

🔹 Corrected Example Sentences
❌ "Local storage stores permanent data to the domain and is accessible in every tab of our application."
✔ "Local storage keeps data even after closing the browser and can be accessed across multiple tabs."

❌ "Session storage stores secure data of our application."
✔ "Session storage holds temporary data and gets cleared when the tab is closed."

*] What is NgRx?
	NgRx is Similar concept to the react redux library they are stored and statmenagement of the data 

	NgRx is a state management library for Angular that follows the Redux pattern.

*] Why do we use NgRx?
	It helps manage state centrally, makes data flow predictable, and improves performance.

*] What are the main parts of NgRx?
	Store: Holds the state.
	Actions: Describe what to do.
	Reducers: Update the state.
	Selectors: Get data from the state.
	Effects: Handle async tasks like API calls.

*] What is NgRx Store?
	A global storage that holds and manages application data.

*] What is an Action in NgRx?
	An Action is an event that tells NgRx what change should happen.

*] What is a Reducer in NgRx?
	A function that takes the current state and an action, then returns a new state.

*] What is a Selector in NgRx?
	A function that helps extract specific data from the store.

*] What are Effects in NgRx?
	Effects handle side effects, like API calls, without blocking the UI.

*] What is the difference between Store and BehaviorSubject?
	Store is immutable and follows Redux.
	BehaviorSubject allows direct updates and does not track history.
	
*] How does NgRx improve performance?
	Uses Selectors to optimize data fetching.
	Uses Immutable State, avoiding direct mutations.
	
What happens if we mutate the state in a reducer?
It breaks the Redux pattern, causing unexpected behavior.

What is the purpose of createSelector()?
It helps get specific data efficiently from the store.

What is the role of Effects in API calls?
Effects listen for actions, call APIs, and dispatch new actions with the results.

What is the benefit of using NgRx Facades?
Facades simplify store interaction, making components easier to manage.

What is an Entity Adapter in NgRx?
It helps manage collections of data efficiently with sorting and filtering.
