## **WPS (Wellington Parcel Service) – Simple Description**

Wellington Parcel Service (WPS) is a parcel delivery platform in Canada. It helps customers create orders and track their shipments. WPS connects customers, businesses, and many third-party carriers in one system.

WPS has **two main portals**:

1. **Order Creation Portal** – Customers create parcel orders here.
2. **Order Tracking Portal** – Customers track their shipments here using a tracking number.

The system uses **role-based access**, so every user can only see the data that belongs to them.

## **Carriers Connected with WPS**

WPS works with many big carriers like:

* Midland
* Obibox
* Frontier

All these carriers send order and tracking information to WPS.

## **Big Businesses Using WPS**

* DHL

These companies use WPS to manage and track thousands of parcels.

## **Main Features**

* Create and manage parcel orders

* Secure login with role-based access

## **How Order Tracking Works (Simple Logic)**

When a parcel moves from one place to another, the carrier sends multiple events.
Example carrier events:

* Order Created
* Picked Up
* In Transit
* Out for Delivery
* Delivered
* Exception (Delay or Issue)

WPS converts these events into **8 main WPS events**, such as:

1. Order Placed
2. Label Created
3. In Transit
4. Out for Delivery
5. Delivered
6. Exception
7. Order Transferred
8. Customer Contacted

### **Tracking Portal Flow**

1. Customer enters tracking number.
2. WPS checks the details from database and carrier API.
3. Carrier events are matched with WPS events.
4. WPS shows a clean timeline like:

   * Order Created
   * Parcel Picked Up
   * In Transit
   * Out for Delivery
   * Delivered

This way, customers can easily check the latest status of their parcel.

14.Explain Memory Leakage in Angular?
	Memory leakage in Angular means:
	Unwanted or no-longer-needed objects are not removed from memory.
	Because they stay in the component or service, Angular cannot free that memory.
	So during application running, these unused objects keep taking space, and this situation is called a memory leak
------------------------------------------------------------------------------------------------------
15.Why Memory Leaks Happens in Angular?
	Memory leaks happen in Angular when unwanted or no-more-needed objects are not removed.
	If subscriptions, events, timers, or data are still running or still stored even after the component is destroyed, then those objects stay in memory.
	Because they are not cleared, they keep taking memory space, and this situation is called a memory leak.
------------------------------------------------------------------------------------------------------
16.How to Detect Memory Leaks in Angular?
	We can detect memory leaks in Angular by checking if the memory is continuously increasing and not getting released.
	We use browser tools to see if unwanted objects, subscriptions, or events are still running even after the component is destroyed.
	If memory keeps growing and old data is not removed, that means memory leak is happening.
------------------------------------------------------------------------------------------------------
17.How to Avoid Memory Leaks in Angular?
	Unwanted or no-longer-needed objects should be removed when they are not needed.
	If we clean these objects properly, we can avoid memory leaks in Angular.

19.What is OnPush in Angular?
->  OnPush is a change detection strategy in Angular.
	Basically, Angular by default detects changes automatically and updates the UI.
	But in this case, unwanted or not-needed changes are also detected, like hovering, clicking, or performing some other action.
	Because of this, change detection runs for the overall application continuously.
	With the help of OnPush, changes are detected only when we want or as per our need, and then the UI is re-rendered accordingly.

25.What is Standalone Component?

->It contains its own imports and dependencies, so it does not depend on any NgModule.
->Standalone components were introduced in Angular 14.
->In the current Angular versions, we can use both standalone components and NgModule-based components, depending on project requirements.

32.Why use are Interceptor?
->Interceptors are used because they work like middleware on the backend. They intercept requests before they reach the server to validate or modify them. If the request is not proper, it can be stopped and not sent to the server.
->Similarly, interceptors also handle responses before they reach the Angular application. If the response is not valid or has errors, it can be handled globally and prevented from affecting the application.

33.I have totoal 10 API , I want to used the interceptor only for 7 API and skip for 3 API , so How can I do this. 
->We have a total of 10 APIs in our application, but we want the interceptor to be applied only to 7 APIs and skipped for the remaining 3 APIs.
To achieve this, we create a skipList that contains an array of API endpoints which should not go through the interceptor.
Inside the interceptor, we write logic to match the request URL with the endpoints present in the skip list. If the request URL matches any of the endpoints, we simply bypass the interceptor logic and forward the request. Otherwise, the interceptor logic is applied as usual.
This approach makes it simple and easy to manage which APIs should skip the interceptor by just updating the skip list.

34.How can you catch error get from backend?
In Angular, we can catch errors returned from the backend using RxJS catchError. This can be done in two main ways.

1️⃣ Request-wise (Service-level) Error Handling

In this approach, we handle errors inside individual service methods using catchError.
This is useful when we want to show custom error messages or apply specific logic for a particular API request.

This approach gives more control because each API can have its own error handling logic.

2️⃣ Global Error Handling using HTTP Interceptor

In this approach, we handle errors globally inside an HTTP interceptor.
This is useful for handling common errors such as authentication failures, network issues, or server errors in one place.

It helps reduce duplicate code and keeps error handling consistent across the application.

📌 Conclusion

Both approaches are commonly used in Angular applications.
Service-level error handling is best for request-specific custom messages, while interceptor-based error handling is best for global and reusable error handling logic.

47.What is the used of @viewChild?

-> @ViewChild is used inside a parent component to access a child component, directive, DOM element, or HTML element from the template.

With the help of @ViewChild, we can:
Call child component methods or access its properties.

Access DOM elements directly

@ViewChild references are usually accessed in the ngAfterViewInit() lifecycle hook, not in ngOnInit().
This is because ngOnInit() runs before the child components or DOM elements are fully initialized, so the @ViewChild value will be undefined.

But in ngAfterViewInit(), all child components and DOM elements are ready, so we can safely use @ViewChild there.
🔁 In short: Use @ViewChild inside ngAfterViewInit() to avoid undefined values.
------------------------------------------------------------------------------------------------------
48.What is used of @viewChildren?
->@ViewChildren works similar to @ViewChild, but @ViewChild is used to access a single child component or element, whereas @ViewChildren is used to access multiple child components or elements at the same time. With @ViewChildren, the parent component can access and update multiple child component properties or call their methods.
------------------------------------------------------------------------------------------------------
Difference between @ViewChild and @ViewChildren
----------------------------------------------------------------------------------------------------------------
| `@ViewChild`                                      | `@ViewChildren`                                          |
| ------------------------------------------------- | -------------------------------------------------------- |
| Used to access **one** child component or element | Used to access **multiple** child components or elements |

74.What is differanance between the Lazy Loading and Eger Loading?
-> Lazy loading and Eager loading are both used to load modules in an Angular application.
In Lazy loading, modules are loaded only when they are needed, meaning when a user navigates to a specific route, only that module is loaded.
In Eager loading, all modules are loaded at the initial application load, even if they are not immediately required.
Lazy loading improves performance and optimizes the Angular application, whereas Eager loading increases initial load time because all modules are loaded at once.
----------------------------------------------------------------
| Lazy Loading                 | Eager Loading                 |
| ---------------------------- | ----------------------------- |
| Loads module **when needed** | Loads **all modules at once** |
| Uses `loadChildren`          | Uses `imports` in `AppModule` |
| Better performance           | Slower initial load           |
| Optimized for large apps     | Suitable for small apps       |

📌 Event Bubbling (Simple Explanation)
🔹 Event bubbling happens when an event on a child element also triggers the same event on its parent elements.
🔹 Example: If you click a button inside a div, the click event first happens on the button, then moves up to the div.

✅ Example of Event Bubbling:

<div onclick="alert('Div Clicked!')">
  <button onclick="alert('Button Clicked!')">Click Me</button>
</div>

👉 When you click the button:

 ️⃣ "Button Clicked!" appears.
 ️⃣ "Div Clicked!" appears next (because of bubbling).

🛑 How to Stop Bubbling?
👉 If you only want to trigger the button click (not the div), use event.stopPropagation().

<div onclick="alert('Div Clicked!')">
  <button id="myButton">Click Me</button>
</div>

<script>
document.getElementById("myButton").addEventListener("click", function(event) {
  event.stopPropagation(); // Stops bubbling
  alert("Button Clicked!");
});
</script>

✅ Now, only "Button Clicked!" appears when you click the button.

🔹 Key Points:
✔ Event bubbling → Events move up from child to parent.
✔ Stops bubbling → Use event.stopPropagation().
✔ Useful for preventing unwanted event triggers.

*]local storage and session storage 
✅ Local Storage
Stores permanent data (until manually cleared).
Data is accessible in all tabs of the same browser.
Common use: storing login tokens, user preferences, themes, etc.
If local storage is cleared, the user is logged out.

✅ Session Storage
Stores temporary data (deleted when the tab or browser is closed).
Data is only accessible in the current tab.
Common use: storing temporary session data like form inputs or session tokens.
More secure than local storage because data is cleared automatically when the session ends.

✅ Common Features
Both can store up to 5MB of data.
Both store data as key-value pairs in the browser.

*] code coverage :- 

👉Code coverage is the concept of checking how much of your code is tested.
👉In Angular, we write unit tests in .spec.ts files to verify our code works correctly.
👉Tools like Jasmine (testing framework) and Karma (test runner) are used to run these tests.
👉Then code coverage tools check which parts of the code are tested and which are not.

Flow (Simple)
	Your Angular Code (.ts files)
			↓
	Unit Tests (.spec.ts files)
			↓
	Run with Jasmine + Karma
			↓
	Generate Code Coverage Report

Example in simple words
	You wrote a service with add() and subtract() functions
	You tested only add() in .spec.ts
	Code coverage shows add() is covered ✅, subtract() not covered ❌
	After writing a test for subtract(), coverage increases

Quick Key Points
	.spec.ts files → Where we write tests
	Jasmine → Testing framework
	Karma → Test runner
	Code Coverage → Shows % of code tested
	Higher coverage = fewer bugs 🚀

🔹 How Does It Work?
 ️⃣ When you create a component in Angular, a test file (component.spec.ts) is automatically created.
 ️⃣ You write test cases in the .spec.ts file to check if your code works correctly.
 ️⃣ To check how much of your code is tested, you run:
	ng test --code-coverage
 ️⃣ This command runs all test cases and stores the results in the /coverage folder.
 ️⃣ You can open the coverage/index.html file to see which parts of your code are tested and which are missing tests.

🔹 Example
If you have a function in a service:

export class MathService {
  add(a: number, b: number): number {
    return a + b;
  }
}

And a test case in math.service.spec.ts:

import { MathService } from './math.service';

describe('MathService', () => {
  let service: MathService;

  beforeEach(() => {
    service = new MathService();
  });

  it('should add two numbers', () => {
    expect(service.add(2, 3)).toBe(5);
  });
});

👉 If you run ng test --code-coverage, it will check if this function is covered in tests.

✅ If tested → It will show a high coverage percentage.
❌ If not tested → It will show lower coverage, meaning you need to write more tests.

🔹 Summary
✔ Code coverage checks if your code is tested properly.
✔ Write tests in .spec.ts files to verify your code.
✔ Run ng test --code-coverage to check coverage.
✔ Higher coverage = Fewer bugs! 🚀

135.Cold(Unicast) vs Hot(Multicast) Observable in RxJS?

In Cold Observable:

	Every subscriber gets data independently and the stream starts fresh for each one.

	It’s like watching a movie on Netflix —
	If 2 or more people watch the same movie,
	each person watches it from the beginning at different times.

 ngOnInit(): void {

    const cold$ = new Observable(observer => {
      observer.next(Math.random());
    });

    cold$.subscribe(val => console.log('Subscriber 1:', val));
    cold$.subscribe(val => console.log('Subscriber 2:', val));

}

In Hot Observable:

	All subscribers receive the same data at the same time.

	It’s like watching a live cricket match —
	Whoever joins will see the current live view,
	and all users see the same moment together.

ngOnInit(): void {
    const subject = new Subject<number>();

    // Both subscribers subscribe before any data is emitted
    subject.subscribe(val => console.log('Subscriber 1:', val));
    subject.subscribe(val => console.log('Subscriber 2:', val));
    subject.subscribe(val => console.log('Subscriber 3:', val));
    subject.subscribe(val => console.log('Subscriber 4:', val));

    // Emit one value
    subject.next(Math.random());
}

*] What is NgRx?
	NgRx is Similar concept to the react redux library they are stored and statmenagement of the data 

	NgRx is a state management library for Angular that follows the Redux pattern.

*] Why do we use NgRx?
	It helps manage state centrally, makes data flow predictable, and improves performance.

*] What are the main parts of NgRx?
	Store: Holds the state.
	Actions: Describe what to do.
	Reducers: Update the state.
	Selectors: Get data from the state.
	Effects: Handle async tasks like API calls.

*] What is NgRx Store?
	A global storage that holds and manages application data.

*] What is an Action in NgRx?
	An Action is an event that tells NgRx what change should happen.

*] What is a Reducer in NgRx?
	A function that takes the current state and an action, then returns a new state.

*] What is a Selector in NgRx?
	A function that helps extract specific data from the store.

*] What are Effects in NgRx?
	Effects handle side effects, like API calls, without blocking the UI.

*] What is the difference between Store and BehaviorSubject?
	Store is immutable and follows Redux.
	BehaviorSubject allows direct updates and does not track history.
	
*] How does NgRx improve performance?
	Uses Selectors to optimize data fetching.
	Uses Immutable State, avoiding direct mutations.
	
What happens if we mutate the state in a reducer?
It breaks the Redux pattern, causing unexpected behavior.

What is the purpose of createSelector()?
It helps get specific data efficiently from the store.

What is the role of Effects in API calls?
Effects listen for actions, call APIs, and dispatch new actions with the results.

What is the benefit of using NgRx Facades?
Facades simplify store interaction, making components easier to manage.

What is an Entity Adapter in NgRx?
It helps manage collections of data efficiently with sorting and filtering.
